# 機能フルレビュー＆ブラッシュアップ プロンプトテンプレート

## 使い方

`[機能名]` を対象機能に置き換えて使用する。
Claude Codeのスキルとしても `/feature-review [機能名]` で呼び出し可能。

---

## 対象機能: [機能名]

### Step 1: 3面調査（並列サブエージェント）

以下の3つの視点で、[機能名]に関わるコードを**全ファイル**洗い出して厳しくレビューしてください。

**エージェントA: 企業管理画面側**
- 管理画面のUI/UX（動線、操作性、プレビュー、エラー表示）
- APIルートの実装品質（N+1クエリ、バリデーション、エラーハンドリング）
- 権限チェック（Owner/Member/Location単位）

**エージェントB: ユーザー（求職者/エンドユーザー）側**
- ユーザー向けUI/UX（受信、表示、アクション、通知）
- メール通知の漏れ（送信されるべきなのに飛んでいないケース）
- ステータス遷移の整合性（不正な状態遷移がないか）

**エージェントC: DB/API/インフラ層**
- DB制約（CHECK、NOT NULL、INDEX）とTypeScript型の整合性
- RLSポリシーの網羅性（SELECT/INSERT/UPDATE/DELETE全部あるか）
- 監査ログの記録漏れ
- SuperAdmin管理画面の有無

### Step 2: 統合レポート

3面の結果を統合し、以下のフォーマットで報告:

| 重要度 | 基準 |
|--------|------|
| Critical | 放置するとデータ不整合・ユーザー体験が壊れる |
| High | 動くけど品質が低い・通知漏れ・パフォーマンス問題 |
| Medium | セキュリティ強化・監査・型整合性 |

### Step 3: Phase分割で修正

- Phase 1: Critical + 簡単なHigh（即修正）
- Phase 2: 残りのHigh + 新規画面
- Phase 3: Medium（セキュリティ・品質）

各Phaseごとに: 実装 → tsc --noEmit → コミット → プッシュ

### 見落とし防止チェックリスト

- [ ] N+1クエリはないか（一覧系APIでループ内にDB呼び出し）
- [ ] DB CHECK制約とTypeScript型は一致しているか
- [ ] 全ステータスのフィルタリングがAPI/UIで可能か
- [ ] メール通知: 全アクション（作成/更新/削除/返信/辞退）で必要な相手に飛ぶか
- [ ] RLSポリシー: SELECT/INSERT/UPDATE/DELETE 全4種あるか
- [ ] 監査ログ: 重要アクションで audit_logs に記録しているか
- [ ] SuperAdmin管理画面: 運営が横断監視できるか
- [ ] Location単位の権限: Memberが担当外を操作できないか
- [ ] フォールバックURL: 本番ドメインになっているか
- [ ] テンプレート変数: 定義と実際の置換が一致しているか
- [ ] Zodスキーマ ↔ DB制約: バリデーションの許可値がDBと同期しているか
