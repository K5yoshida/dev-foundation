# プロダクト確信度チェック

> Input: 検討中の機能、または「定期チェック」
> Output: 確信度スコア + 判定（GO / PIVOT / KILL）+ CONVICTION_LOG 記録
> 所要時間: 30分〜1時間
> タイミング: スペック作成前、月2回の定期実行、方向性に迷ったとき

---

## このプロンプトの位置づけ

プロダクト開発は3層構造になっている:

```
WHO（誰に）× WHAT（どんな価値を）= 売れるか？
       ↓
市場規模 × 競合性 × モート（防御壁）= 維持・拡大できるか？
       ↓
HOW = プロダクト（機能・画面・コード）← 他の20個のプロンプトが扱う範囲
```

**このプロンプトだけが上2層を扱う。**
HOW をいくら精密に作っても、WHO × WHAT が間違っていれば全てゴミ。

---

## Phase 1: ビジネス構造の確認

設計書からプロダクトのビジネス構造を読み取る。

### 読むべき設計書

CLAUDE.md の「設計書クイックリファレンス」を参照し、以下に該当するドキュメントを特定して読む:

| 探すもの               | 記載されている内容                                           |
| ---------------------- | ------------------------------------------------------------ |
| **ビジョン・戦略**     | プロダクトの存在意義、偉大な目標                             |
| **市場分析・価値提案** | WHO（ターゲット）、WHAT（提供価値）、4つの壁（解決する課題） |
| **競合分析**           | 競合との差別化ポイント、モートの定義                         |
| **収益モデル**         | 料金体系、ユニットエコノミクス、成長レバー                   |
| **GTM戦略**            | 顧客獲得チャネル、パートナー戦略                             |
| **リスク分析**         | 上位リスクとその対策                                         |

### 抽出するもの（WHO × WHAT マトリクス）

```
WHO:
  - プライマリターゲット: [設計書から抽出]
  - セカンダリターゲット: [設計書から抽出]
  - 各ターゲットの課題: [設計書から抽出]

WHAT:
  - コア価値提案: [設計書から抽出]
  - 差別化ポイント: [設計書から抽出]
  - 成長レバー: [設計書から抽出]

HOW（検討中の機能）:
  - 機能名: [入力された機能]
  - WHO のどの課題を解決するか: [分析]
  - WHAT のどの価値を実現するか: [分析]
```

## Phase 2: 市場の現実チェック

### 2.1 競合動向（AI実行可能）

設計書の競合分析セクションに記載された競合を、Web検索で最新状況を確認する。

チェックポイント:

- 競合の直近3ヶ月のアップデート・新機能
- 新興の競合が出現していないか
- 設計書に書かれた差別化ポイントが今も有効か

```
検索例:
- "{競合名} 新機能" or "{競合名} アップデート" + 直近3ヶ月
- "{プロダクトカテゴリ} SaaS 新サービス"
```

### 2.2 顧客・チャネルの声（人間アクション候補）

AIが自律で実行できない項目は「人間アクション候補」として出力する。
実行するかどうかはオーナーが判断する。

```markdown
## 人間アクション候補

- [ ] ターゲット顧客3名に15分ヒアリング:
      「今一番困っていることは？」「この機能あったら使いますか？」
- [ ] チャネルパートナー（いる場合）に確認:
      「これがあれば顧客に提案しやすくなりますか？」
- [ ] 直近の解約理由・離脱データを確認
- [ ] アクティブユーザーの利用パターンを確認（どの機能を使っている/使っていない）
```

## Phase 3: 5つの根本質問

検討中の機能について、WHO × WHAT ベースで回答する。

| #   | 質問                                                                                                                   | 回答 | 確信度(1-5) |
| --- | ---------------------------------------------------------------------------------------------------------------------- | ---- | ----------- |
| Q1  | **WHO接続**: この機能は、設計書に定義されたターゲットのどの課題を解決するか？課題と機能の因果関係を1文で説明せよ       |      |             |
| Q2  | **WHAT接続**: この機能は、設計書に定義された価値提案のどれを実現するか？実現しない価値提案は「対象外」と明記せよ       |      |             |
| Q3  | **モート評価**: 競合がこの機能を真似するのに何ヶ月かかるか？6ヶ月以下なら差別化にならない。それでも必要な理由は何か？  |      |             |
| Q4  | **目標必須性**: この機能がなくても設計書に定義された目標（売上・契約数・市場シェア等）を達成できるか？                 |      |             |
| Q5  | **Ship翌日テスト**: 明日リリースしたら、ターゲット顧客に「使ってください」と自信を持って言えるか？言えない理由は何か？ |      |             |

### 確信度の基準

| スコア | 基準                                     |
| ------ | ---------------------------------------- |
| 5      | 確実。データや顧客の声で裏付けがある     |
| 4      | 高い。論理的に筋が通り、設計書と整合する |
| 3      | 普通。妥当だが検証が足りない             |
| 2      | 低い。推測に依存している                 |
| 1      | 不明。根拠がない                         |

### 判定

| 合計   | 判定      | アクション                                 |
| ------ | --------- | ------------------------------------------ |
| 20〜25 | **GO**    | スペック作成に進む                         |
| 15〜19 | **PIVOT** | 方向性を微調整してから進む。調整内容を明記 |
| 14以下 | **KILL**  | この機能は作らない。別の機能を検討         |

## Phase 4: 優先順位の再評価

現在の実装候補リストを「WHO × WHAT への貢献度」で再ランキングする。

| 順位 | 機能 | WHO接続 | WHAT接続 | 確信度 | 判定 |
| ---- | ---- | ------- | -------- | ------ | ---- |
| 1    |      |         |          |        |      |
| 2    |      |         |          |        |      |
| ...  |      |         |          |        |      |

### ランキング基準

1. **WHOの課題を直接解決するもの** → 最優先
2. **成長レバーに直結するもの**（チャネル拡大、ネットワーク効果等）→ 次優先
3. **運用基盤として必要なもの** → 必要だが後回し可能
4. **WHO × WHAT に接続しないもの** → KILL

---

## Phase 5: CONVICTION_LOG 記録

`.agent/shared/CONVICTION_LOG.md` に以下を追記:

```markdown
## YYYY-MM-DD

### 対象: {機能名}

### 確信度: {合計点}/25

### 判定: GO / PIVOT / KILL

### WHO接続: {ターゲットのどの課題を解決するか}

### WHAT接続: {どの価値提案を実現するか}

### 市場変化: {競合動向で気づいたこと}

### 人間アクション: {オーナーがやるべきこと}
```

## 完了条件

- [ ] Phase 1: ビジネス構造（WHO × WHAT）を設計書から抽出済み
- [ ] Phase 2: 競合の最新動向を確認済み
- [ ] Phase 3: 5つの根本質問に全て回答（確信度スコア付き）
- [ ] 判定が GO / PIVOT / KILL のいずれかで確定
- [ ] PIVOT → 調整方針が明記されている
- [ ] KILL → 代替候補が提示されている
- [ ] CONVICTION_LOG に記録済み
- [ ] 人間アクション候補が明示されている

## 禁止

- 確信度を「なんとなく高く」つけること（根拠を1文で書けないなら3以下）
- 全てを GO にすること（KILL する勇気がないのは最大のリスク）
- 「便利そう」「あると嬉しい」を根拠にすること（WHO × WHAT で語れ）
- 設計書を読まずに回答すること（設計書がビジネス構造の唯一の正）
