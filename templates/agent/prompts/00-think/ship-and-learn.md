# Ship後 学習回収

> Input: 直近リリースした機能のスペック番号
> Output: 学習記録 + 次アクション + LEARN_LOG 記録
> 所要時間: 30分
> タイミング: 機能リリース後 48時間〜1週間以内（必須）

---

## なぜ必要か

機能を作って終わりは、「テストを受けて答え合わせしない」のと同じ。
WHO × WHAT への接続が実際に機能したかを確認しなければ、次も同じ精度でしか作れない。

```
仮説（spec.md）→ 実装 → Ship → 検証（このプロンプト）→ 学習 → 次の仮説
                                    ↑ ここが抜けると成長しない
```

---

## Phase 1: 仮説の答え合わせ

spec.md の **Business Impact Hypothesis** セクションを開き、仮説と現実を突き合わせる。

| 仮説                              | 予測値 | 実測値 | 差分 | 考察 |
| --------------------------------- | ------ | ------ | ---- | ---- |
| WHO接続（ターゲットの課題解決）   |        |        |      |      |
| WHAT接続（価値提案の実現度）      |        |        |      |      |
| 独自指標（spec.mdで定義したもの） |        |        |      |      |

### データの取得方法

**AI実行可能な範囲**:

- ダッシュボード/アナリティクスAPIからデータを取得
- DB から リリース前後のイベントデータを比較
- エラーログの確認（機能がエラーなく動いているか）

**データが取れない場合**:

- 計測コードが入っていない → 次スプリントで計測追加を最優先タスクにする
- まだ利用者がいない → 人間アクション: ターゲット顧客に「使ってみてください」と連絡
- 期間が短すぎる → 1週間後に再実行をスケジュール

---

## Phase 2: ユーザー行動の確認

### 2.1 定量データ（AI実行可能）

- この機能の画面/APIのアクセス数
- 完了率（機能を最後まで使い切ったユーザーの割合）
- エラー発生率
- リリース前後の主要KPIの変化

### 2.2 定性フィードバック（人間アクション候補）

```markdown
## 人間アクション候補

- [ ] ターゲット顧客1名以上に「新機能使いましたか？どうでした？」と聞く
- [ ] 「使わなかった」場合は理由を聞く（知らない？不要？使いにくい？）
- [ ] チャネルパートナー（いる場合）に「この機能、顧客に提案しましたか？」と聞く
```

---

## Phase 3: 学習の抽出

### 何がわかったか

1. WHO × WHAT の仮説は正しかったか？
2. 予想外の使い方はあったか？
3. 次に作るべきものの優先順位は変わったか？

### 3つの判定

| 判定        | 基準                         | 次アクション                                                       |
| ----------- | ---------------------------- | ------------------------------------------------------------------ |
| **HIT**     | 仮説通りまたはそれ以上の効果 | 横展開・深掘りを検討。WHO × WHAT の接続が正しいことが実証された    |
| **MISS**    | 仮説より効果が低い           | 原因分析。WHO接続が間違い？WHAT接続が間違い？HOW（実装）が悪い？   |
| **UNKNOWN** | データ不足で判定不能         | 計測追加が最優先。判定できない状態でコードを追加するのは資源の浪費 |

### MISS の場合の深掘り

| 原因の層 | 質問                                   | 対応                               |
| -------- | -------------------------------------- | ---------------------------------- |
| **WHO**  | ターゲット自体が間違っていないか？     | → product-conviction でWHOを再検討 |
| **WHAT** | 価値提案がターゲットに響いていないか？ | → 設計書のWHATセクションを見直し   |
| **HOW**  | 実装が悪いだけで、方向性は正しいか？   | → 改善タスクを追加して修正         |

---

## Phase 4: LEARN_LOG 記録

`.agent/shared/LEARN_LOG.md` に以下を追記:

```markdown
## YYYY-MM-DD

### 対象: {機能名 or specs/{number}-{name}}

### リリース日: YYYY-MM-DD

### 判定: HIT / MISS / UNKNOWN

### 仮説: {元の Business Impact Hypothesis}

### 現実: {実測データ}

### 学習: {1〜3行。次に活かせる知見}

### 原因層: {WHO / WHAT / HOW のどこに問題があったか}

### 次アクション: {具体的なタスク}

### 優先順位の変更: {あれば記載}
```

---

## Phase 5: フィードバックをプロダクトに戻す

学習をプロダクトの次のサイクルに反映する。

| 判定       | アクション                                                     |
| ---------- | -------------------------------------------------------------- |
| HIT        | 横展開のスペック作成を検討（product-conviction → spec-create） |
| MISS(WHO)  | product-conviction でターゲットを再検討                        |
| MISS(WHAT) | 設計書のビジョン・価値提案セクションを更新（docs-sync）        |
| MISS(HOW)  | 改善タスクを追加。polish-product or plumbing-ux-flow で修正    |
| UNKNOWN    | 計測タスクを最優先で追加。計測なしに次の機能を作るな           |

---

## 完了条件

- [ ] 仮説と実測の突き合わせ完了
- [ ] 判定（HIT / MISS / UNKNOWN）確定
- [ ] MISS の場合: 原因層（WHO / WHAT / HOW）が特定されている
- [ ] LEARN_LOG に記録済み
- [ ] 次アクションが明示されている
- [ ] 優先順位に変更が必要なら IMPLEMENTATION_STATUS.md を更新

## 禁止

- リリースして満足すること（Ship は始まりであって終わりではない）
- データなしで「うまくいった」と判断すること
- 学習を記録せずに次の機能に進むこと
- UNKNOWN のまま放置すること（計測できない = 成長できない）
