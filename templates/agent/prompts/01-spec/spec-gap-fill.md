# スペック横断整合性チェック

> Input: なし（`specs/` 全体を走査）
> Output: 横断的な抜け漏れ・矛盾リスト + 修正
> 所要時間: 1〜2時間
> 前提: 複数のP0スペックが作成済みであること

---

## 手順

### Step 1: 全スペック読み込み

`specs/*/spec.md` を全て読み込み、以下を抽出:

- 各スペックの Data Model Changes（テーブル/カラム）
- 各スペックの API Changes（エンドポイント）
- 各スペックの Acceptance Criteria
- 各スペック間の依存関係（「スペックAの機能がスペックBの前提」等）

### Step 2: 依存関係マトリクス作成

```
| スペック | 依存先 | 共有テーブル | 共有API | 共有コンポーネント |
```

### Step 3: 矛盾・抜け漏れ検出

1. **DB矛盾**: 同じテーブルに対して異なるカラム定義をしていないか
2. **API矛盾**: 同じエンドポイントに対して異なるリクエスト/レスポンス定義をしていないか
3. **依存未定義**: 「AがBに依存」だがBのスペックにそれが書いていない
4. **共通基盤の漏れ**: 複数スペックが同じ共通処理を必要としているのに、誰も作る担当になっていない
5. **優先順位の矛盾**: 依存先スペックより依存元スペックが先に実装される計画になっていないか

### Step 4: 修正

発見した問題を各 spec.md に反映する（自然な統合で、追記の痕跡を残さない）。

## 完了条件

- [ ] 依存関係マトリクス作成済み
- [ ] 矛盾 0件
- [ ] 未定義の依存 0件
- [ ] 共通基盤の担当が明記済み
