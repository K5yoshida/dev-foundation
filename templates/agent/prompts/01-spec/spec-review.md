# スペック＆設計書レビュー

> Input: レビュー対象（`specs/{number}/` または `docs/{number}`）
> Output: 弱点リスト + 精密化した修正
> 所要時間: 1〜2時間
> モード: プランモード推奨

---

## 2つのレンズで見る

### レンズ1: 1mm精度チェック（詳細検証）

エンジニアが「何を実装すればいいかわからない」箇所を全て洗い出す。

チェック観点:

1. **画面仕様**: 各ボタン・入力・表示要素の挙動が1つ残らず定義されているか
2. **状態遷移**: 全ての状態とその遷移条件が網羅されているか（Empty/Loading/Error/Success）
3. **エッジケース**: 0件、上限値、同時操作、タイムアウト、権限境界が定義されているか
4. **データフロー**: ユーザー操作 → API → DB → 画面反映の全経路が追跡可能か
5. **受入基準**: 全て「Yes/Noで判定可能」か（曖昧な形容詞がないか）

### レンズ2: ゼロベース再考（構造検証）

「もしゼロから設計し直すとしたら、同じ設計にするか？」を問う。

問いかけ:

1. この機能の本質的な価値は何か？それは設計に反映されているか？
2. ユーザーの業務フロー上、この機能はどこに位置するか？前後の文脈は考慮されているか？
3. 競合と比較して、この設計で「指名検索CVR運用OS」としての優位性が出るか？
4. 3ヶ月後、この設計で運用が回るか？パートナーが複数顧客に横展開できるか？
