# コードレビュー

> Input: レビュー対象のファイル範囲（git diff、PR、または指定ファイル）
> Output: 重大度別の指摘リスト + 修正（承認後）
> 所要時間: 30分〜1時間

---

## 12の観点

| #   | 観点                | チェックポイント                                                           |
| --- | ------------------- | -------------------------------------------------------------------------- |
| 1   | UX/ユーザビリティ   | エラーメッセージの分かりやすさ、レスポンス速度、操作の一貫性               |
| 2   | UI/デザインシステム | デザイントークンの一貫性、コンポーネントライブラリとの整合性、レスポンシブ |
| 3   | アクセシビリティ    | WAI-ARIA、キーボードナビ、コントラスト比                                   |
| 4   | 事業目的との整合性  | 要件の充足度、ビジネスロジックの正確性                                     |
| 5   | システム設計        | 責務分離、依存関係、拡張性・保守性                                         |
| 6   | セキュリティ        | 認証・認可、入力検証、SQLi/XSS/CSRF対策                                    |
| 7   | パフォーマンス      | N+1問題、不要な再レンダリング、バンドルサイズ                              |
| 8   | 機能要件            | 仕様充足度、境界値・異常系の処理                                           |
| 9   | データ設計          | スキーマ、インデックス、整合性制約                                         |
| 10  | コード品質          | 命名規則、DRY、型安全性、既存パターンとの一貫性                            |
| 11  | テスタビリティ      | テストの書きやすさ、モック可能な設計                                       |
| 12  | 運用性              | ログ出力、エラートラッキング、グレースフルデグラデーション                 |

## 出力

### 総合評価: A/B/C/D/E

| 評価 | 基準                               |
| ---- | ---------------------------------- |
| A    | 本番リリース可能。軽微な改善点のみ |
| B    | 小規模な修正後にリリース可能       |
| C    | 中規模な修正が必要                 |
| D    | 大規模なリファクタリングが必要     |
| E    | 設計からの見直しが必要             |

### 指摘事項（重大度順）

各指摘に含めるもの:

- ファイル:行番号
- 問題の説明
- 推奨対応
- コード例（該当する場合）

## 修正フロー

1. レビュー結果を報告
2. 承認を得てから修正着手
3. 修正後に `npm run build && npm run test && npm run lint` で検証
4. 修正完了報告
