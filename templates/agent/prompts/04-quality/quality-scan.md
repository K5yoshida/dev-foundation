# 品質スキャン（守り）

> Input: なし（プロジェクト全体を走査）
> Output: 5Phase の品質レポート + 修正
> 所要時間: Phase 1のみ=1時間 / 全Phase=3〜5時間

---

## 実行方法

| タイミング         | 実行範囲     | 所要時間 |
| ------------------ | ------------ | -------- |
| 機能追加の後       | Phase 1 のみ | 1時間    |
| リリース前         | Phase 1 + 3  | 2時間    |
| 月1定期点検        | 全Phase通し  | 3〜5時間 |
| 事業指標が伸び悩み | Phase 4 集中 | 2時間    |

## Phase 1: ハリボテ検出

1. **全ページの4状態チェック**: Empty / Loading / Error / Success が全て実装されているか
   - `app/(dashboard)/*/page.tsx` を走査し、loading.tsx / error.tsx の有無を確認
   - 空データ時の表示（EmptyState コンポーネント）の有無を確認
2. **Error Boundary**: 各セクションに配置されているか
3. **APIエラーハンドリング**: 全APIルートで統一フォーマットか。空catchブロック禁止

## Phase 2: テストで証明

1. `npm run test:coverage` でカバレッジ計測
2. テストがないAPIルートを特定 → 統合テスト追加（200/401/400/404の4パターン）
3. 目標: 80%以上

## Phase 3: 磨き込み

1. **Friction Log**: 主要ユーザーフロー（サインアップ→バナー作成→公開→CV確認）を通して引っかかりを記録
2. **マイクロインタラクション**: ボタンのローディング状態、成功/エラーフィードバック、トースト
3. **コピー品質**: ターゲットユーザー（Webに疎い中小事業者）が理解できる言葉か

## Phase 4: 事業逆算

1. **North Star Metric** の定義（設計書のビジネス戦略ドキュメントから特定）
2. **オンボーディング完走率**: サインアップからゴールまでの各ステップの通過率
3. **Activation指標**: ユーザーが「Aha!」と感じる瞬間の計測
4. **LEARN_LOG参照**: `.agent/shared/LEARN_LOG.md` の直近学習（HIT/MISS/UNKNOWN）を踏まえて、事業指標の評価基準を更新する

## Phase 5: 運用品質

1. **パフォーマンス**: LCP, CLS（Lighthouse計測）
2. **セキュリティ**: `npm audit --production`
3. **アクセシビリティ**: Lighthouse a11y 90+目標
