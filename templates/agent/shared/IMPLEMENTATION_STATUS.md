## 1. 現状認識

現行リポジトリには、以下の「土台」はすでにある。

- 認証
- テナント/パートナーの基本構造
- バナー、LP、モーダル、LINE の基本導線
- 公開 widget 配信
- 基本的なイベント計測
- 管理画面の骨格
- 一部の分析画面

しかし、これは「カテゴリを勝ち切るプロダクト」の完成ではない。

v2 の基準で重要なのは、以下である。

1. 来訪者文脈に応じて 3CV を自動で選べるか
2. 導入前後で成果を証明できるか
3. 改善ループが回るか
4. パートナーが複数顧客へ横展開できるか
5. 本番運用で壊れても即検知できるか

この基準では、現状は **20〜30%** と評価する。

## 8. 外部設定の現実（2026/03/04 確認時点）

### 8.1 Supabase

- マイグレーションは local / remote で整合している
- ただし、ローカルスタック稼働は別問題であり、Docker 未起動時はローカル確認不可

### 8.2 Vercel

- Stripe / Supabase 系の主要 env は投入済み
- `CRON_SECRET` は Production に投入済み
- `npm run audit:external` で、placeholder を除いた runtime-valid 判定でも、必須欠損は `SENDGRID_API_KEY` / `SENDGRID_FROM_EMAIL` の 2 つだけと確認済み
- `SENDGRID_FROM_NAME` は未設定でも既定値へフォールバックするため、監査上は optional fallback 扱い
- cron 3 本は `dry_run=1` で副作用なし確認が可能
- production には最新コードを反映済みで、`smoke:prod` 上でも cron 3 本の未認証 guard は `401` で通る
- production env を一時 pull した確認でも、cron 3 本の認証付き `dry_run` はすべて `200` で通る
- 管理者の売上画面 API に運用ステータスを追加し、SendGrid 未設定などの残ブロッカーを UI から確認できる
- `ops_status` では `missingRequiredEnv` / `optionalFallbackEnv` を返し、足りない env 名をそのまま確認できる
- `ops_status.featureStatus` では、CV通知メール / 週次ダイジェスト / Cron / 報酬バッチの稼働可否を機能単位で確認できる
- この管理者向け `ops_status` 追加も production へ反映済み
- `smoke:prod` は `/api/admin/revenue` を含めて `10 checks passed`、production env を一時 pull した full smoke は `13 checks passed`
- full smoke（`SMOKE_ENV_FILE` 付き）は `ops_status.featureStatus` も必須検証する
- SendGrid の設定判定は `getSendGridSetupStatus()` に共通化し、管理画面 / weekly-digest / 通知処理で同じ不足 env を参照する
- `weekly-digest` は SendGrid 未設定時、`createAdminClient()` や tenant scan の前に停止し、不足 env を返す
- route 契約自体は `tests/api/admin-revenue-route.test.ts` で固定済み
- `SENDGRID_*` は未投入のため、メール送信系は本番で一部未稼働

### 8.3 Stripe

- ルートと基本接続はある
- ただし、「Price 作成 / Portal 設定」はコード完了ではなく、運用確認タスクとして残る

### 8.4 本番運用

- CloudFlare CDN
- カスタムドメイン
- 監視 / アラート

は、v2 で重要な割に未完成

## 9. 今後の判定ルール

以後、以下は「完了」の根拠に使わない。

- 画面が存在する
- API が存在する
- 単体テストが存在する
- 旧 v1 タスクリストを消化している

以下を満たしたときだけ、進捗を前に進める。

1. 事業価値に直結する
2. 計測できる
3. 証明できる
4. パートナー運用に耐える
5. 本番で監視できる
6. `docs/31_実装着手前チェックリスト.md` が埋まり、実装開始の判断根拠が残っている
7. 実装と同時に更新すべき docs が反映されている

---

## 10. 結論

現状は「終盤」ではない。

現状は、

- 見える機能の土台がある
- しかし、勝つための心臓部がまだ薄い

という状態である。

したがって、進捗管理の前提を「v1 の残タスク消化」から、
**「v2 の中核能力をどこまで作れたか」**
へ完全に切り替える。
