# 5 段階リスク分類 (L0〜L4)

## 概要

全ての操作を「不可逆性」と「影響範囲」で 5 段階に分類し、承認フローをレベル別に設計する。

**原則**: 分類が曖昧なら 1 段上を採用する。

---

## 各レベル詳細

### L0 🔴 IRREVERSIBLE (取り返しがつかない)

**特徴**: 元に戻せない、または戻すのに多大なコストがかかる

**例 (DB 開発系プロジェクト)**:
- 本番 DB の `DELETE` / `TRUNCATE`
- 本番マイグレーション適用 (特に DROP TABLE / DROP COLUMN)
- `git push --force` を main ブランチに
- 顧客へのメール一括送信
- 決済処理の実行
- 外部 API の不可逆呼び出し (Stripe charge 等)

**例 (コンサル / ドキュメント系プロジェクト)**:
- 顧客への提案書・契約書の最終版送付
- 社内全体への重要アナウンスメール送信
- 外部向けプレスリリース公開
- 契約書のハンコ押印・電子署名
- 予算の確定申告・経営会議決議
- クラウドリソースの削除 (S3 bucket / EC2 インスタンス等)

**承認フロー**:
- 対面協議 必須
- プロジェクトオーナー本人が実行 (AI は提案のみ)
- 週最大 2 回まで
- 事前 dry-run + rollback 計画書必須

### L1 🟠 HIGH RISK (高リスク)

**特徴**: ミスすると本番影響大、ただし修正可能

**例 (DB 開発系プロジェクト)**:
- 本番 DB の `UPDATE` (数万行規模)
- staging → 本番への昇格
- 新規マイグレーションの作成 (適用はまだ)
- 依存ライブラリのメジャーバージョン変更
- CI/CD パイプラインの変更
- API エンドポイントの破壊的変更

**例 (コンサル / ドキュメント系プロジェクト)**:
- 提案書・契約書のドラフト最終版 (送付前最終レビュー)
- 社内向け重要通達の下書き承認
- 仕様書の大幅改訂 (関連ステークホルダーへ影響)
- 業務フロー変更の承認
- ベンダー契約の条件変更

**承認フロー**:
- 個別事前承認 必須 (プロジェクトオーナーが明示的に「OK」)
- **24h 滞留** (作成から 24 時間待ってから適用)
- **週 2 件上限**
- Red Team 3 並列 + Codex 2 並列レビュー必須
- Critical 0 + High 0 まで繰り返しレビュー

### L2 🟡 MID RISK (中リスク)

**特徴**: 本番には直接影響しない、またはロールバック容易

**例**:
- コードの機能追加・修正
- ローカル DB でのマイグレーション実行
- staging DB の INSERT / UPDATE
- E2E テストの追加
- UI の非破壊的変更

**承認フロー**:
- Phase 着手時に一括承認を取る
- 各操作の事後報告は不要 (サマリー報告のみ)
- Red Team レビューは推奨 (必須ではない)

### L3 🟢 LOW (低リスク)

**特徴**: 戻せる、影響範囲が限定的

**例**:
- ドキュメントの更新
- `memory/` ファイルの更新
- サンプル 100 件以下の SELECT クエリ
- ログファイルの閲覧
- ローカル環境の設定変更

**承認フロー**:
- 事後報告のみ (セッション終了時にまとめて)
- 承認不要

### L4 ⚪ READ ONLY (読み取り専用)

**特徴**: 一切の書き込みなし

**例**:
- `Read` ツールでのファイル閲覧
- `Grep` / `Glob` での検索
- `git log` / `git status`
- DB の `SELECT` (大規模でも OK、書き込みなし)
- API の GET リクエスト

**承認フロー**:
- 無制限、自由
- 報告不要

## 運用ルール

### 冒頭宣言

各操作を始める前に「これは Lv X です」を宣言する。

**良い例**:
```
「これは L1 操作です。本番 DB の users テーブルに UPDATE を行います。
24h 滞留済み、プロジェクトオーナーの承認を取ってから適用します。」
```

**悪い例**:
```
「マイグレを適用します」  ← レベル宣言なし、承認フロー不明
```

### 曖昧な時は 1 段上

- 「コード変更だけど影響範囲が広い」→ L2 ではなく L1 扱い
- 「memory 更新だけど他プロジェクトに影響する」→ L3 ではなく L2 扱い

### L0 / L1 の実行制限

- L0: 週 最大 2 回
- L1: 週 2 件上限、24h 滞留必須
- 上限超過時は翌週に繰り越し

### 緊急例外

本番障害等で即対応が必要な場合:
1. プロジェクトオーナーに「緊急例外プロトコル発動」を宣言
2. 24h 滞留をスキップ
3. RISK_REGISTER に例外発動を記録
4. 次週の L1 枠から 1 件減らす (補償)

## よくある誤解

### Q: 「Read だけだから L4」と考えていいか?

A: 基本 OK。ただし**機密情報を含むファイル**を読む場合は L3 扱いにして記録を残すこともある (例: 顧客データ・医療記録)。

### Q: マイグレーションファイルの作成は L1 か L2 か?

A: **L1**。ファイル作成自体は書き込みだが、内容が本番 DB に後で適用されるため、作成時点で設計欠陥があると本番事故に直結する。24h 滞留中に再レビューする意味がある。

### Q: git commit は何レベル?

A: **L2〜L3**。commit 自体は local だが、ファイルによって影響が変わる:
- memory / docs 変更のみ → L3
- コード変更 → L2
- マイグレーション追加 → L1 (上記理由)

### Q: git push origin main は?

A: **L2** (通常)。ただし Vercel 自動デプロイ等で本番反映される場合は **L1** に格上げ。

## 他フレームワークとの比較

| フレームワーク | リスク分類 |
|---|---|
| COF | L0〜L4 の 5 段階 |
| ITIL | RFC (Request for Change) の Standard / Normal / Emergency |
| SRE | SLO / Error Budget ベース |
| Anthropic Claude Code | Hook の PreToolUse / Stop 等の event ベース |

COF は**操作ベース**の分類であり、承認フローと直接結びつくのが特徴。
