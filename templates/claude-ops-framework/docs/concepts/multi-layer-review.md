# 多層 Red Team レビュー

## 問題

単一モデルの自己レビューは**盲点を見落とす**。Claude に「このコードをレビューして」と
頼んでも、同じモデルが書いて同じモデルがレビューするため、同じ思考の癖でチェックが
甘くなる。

## 解決策: 敵対的 + 異質モデル

```
┌─────────────────────────────────────────┐
│  初版コード/マイグレを Claude が作成       │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  層 1-A: Claude 敵対的サブエージェント 3 並列  │
│  - Red Team (攻撃側視点)                │
│  - Fact Check (実データ検証)            │
│  - Incident Sim (事故再現)              │
└─────────────────────────────────────────┘
                  ↓ Critical/High 検出
┌─────────────────────────────────────────┐
│  層 1-B: Codex (GPT-5 系) 異質モデル 2 並列 │
│  - 異なる思考パターンで再検証             │
│  - Claude が見落とした欠陥を拾う          │
└─────────────────────────────────────────┘
                  ↓ 指摘全反映
┌─────────────────────────────────────────┐
│  Round 2 再レビュー                     │
│  ↓ 残指摘全反映                         │
│  Round 3 再レビュー                     │
│  ↓ ...                                  │
│  Round N: Critical 0 + High 0 到達      │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  層 2: 盲検テスト (該当時)                │
│  - 500 件層別サンプルで精度検証          │
└─────────────────────────────────────────┘
                  ↓
┌─────────────────────────────────────────┐
│  層 3: 業務受入 (プロジェクトオーナー本人)  │
└─────────────────────────────────────────┘
```

## ラウンド進行の典型パターン

```
Round 1: Critical 0 / High 4 / Medium 4 / Low 3   ← 初版の欠陥
  ↓ 全反映
Round 2: Critical 0 / High 2 / Medium 1 / Low 1   ← 新規検出
  ↓ 全反映
Round 3: Critical 0 / High 0 / Medium 1 / Low 0   ← 収束
  ↓ コミット可
```

## Codex (異質モデル) が必須の理由

Claude 同士のレビューは、例えば以下の盲点を共有する:
- 同じ「安全そうに見える」パターンを両方とも安全と判定
- 同じ API の使い方を両方とも正しいと思い込む
- 同じ用語の誤解を両方とも見逃す

Codex (GPT-5 系) は別の訓練データ・別のアーキテクチャで動くため、
**Claude が見落とした別角度の欠陥**を拾う確率が高い。

MEDICA SOERUTE の Project Apex では、以下のような Codex 固有の発見があった:
- 「v11 migration breaks fresh database replay」 (Claude は本番適用を前提に設計、fresh を想定していなかった)
- 「fresh replay produces schema/type drift」 (Claude は CHECK 制約だけ追加、schema parity を見落とし)

## リワーク回数上限

- **3 回まで** が目安 (README・CLAUDE.md.append と統一)
- 4 回目以降はスコープ縮小を検討 (設計そのものを見直す方が早い)
- Critical 0 + High 0 を無限に追い求めると完璧主義の罠

### スコープ縮小の具体手順

4 ラウンド目以降に High 以上が残った場合:

1. **対象件数を半減**: 例えば 786 件降格 → 上位 100 件のみに縮小、残りは次回 L1 枠へ
2. **ファイル単位で分割**: 1 マイグレで全 Phase 統合 → Phase 2-3 と Phase 5-7 を別マイグレに
3. **機能単位で分離**: A 機能 + B 機能を統合実装 → A を先行 Phase、B を次 Phase に
4. **リスクレベルを下げる**: L1 マイグレを L2 staging 動作確認に格下げして知見蓄積
5. **プロジェクトオーナーに協議**: 上記 1-4 で対処できない場合、スコープ自体の再定義

スコープ縮小は「失敗」ではなく、Apex 最上位原則「間違っていないこと > 充填率」
に沿った正しい判断です。

## Codex Stop hook との関係

Claude Code の Stop hook に Codex 自動レビューを組み込むと、
**セッション終了時に自動で最終チェック**が走る。

Critical 指摘が出たら即対応し、次セッションに持ち越さない。

## レビュー分類の基準

### Critical

- 本番データ破損の可能性
- セキュリティ脆弱性
- コミット不可レベル

### High

- 設計欠陥 (機能は動くが構造問題)
- 重要ケースでの失敗
- コミット強く非推奨

### Medium

- 運用上の注意点
- 将来のメンテナンス困難さ
- コミット可だが反映推奨

### Low

- コメント・命名・ドキュメント
- 軽微な改善余地
- コミット可

## Claude Red Team サブエージェントの役割分担

### Red Team (攻撃側視点)

「このコードを壊すにはどうすればいい?」を考える。
- 悪意ある入力
- エッジケース
- 並行処理の race condition
- 想定外の実行順序

### Fact Check (実データ検証)

「主張されている数値は正しいか?」を検証。
- 件数・サイズの実測
- API レスポンスの実測
- ドキュメント vs 実装の乖離

### Incident Sim (事故再現)

「過去事故と同型のパターンはあるか?」を探す。
- 単一キー突合
- 無検証の統合
- 上書き事故の温床

## Codex quota 切れ時の代替

Codex API quota が切れた時は、Claude Red Team を**+1 ラウンド**実施して代替する。
ただし同一モデル内のレビューなので、100% の代替にはならない点に注意。

長期プロジェクトでは Codex quota 管理も運用計画に含めるべき。
