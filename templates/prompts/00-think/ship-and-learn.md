# Ship後学習ループ

> Input: リリースした機能名
> Output: HIT / MISS / UNKNOWN 判定 + LEARN_LOG 記録 + 次アクション
> 所要時間: 30分
> タイミング: 機能リリース後 48時間〜1週間

---

## WHY

作りっぱなしは最大の無駄。Ship後に WHO × WHAT への接続が実際に機能したか検証し、学習をプロダクトに戻す。

---

## Phase 1: 仮説の答え合わせ

`specs/{機能名}/spec.md` の Business Impact Hypothesis セクションを読む。

| 項目 | 仮説（spec.mdから） | 実測値 | 差分 |
|------|---------------------|--------|------|
| 指標1 | | | |
| 指標2 | | | |
| 指標3 | | | |

## Phase 2: ユーザー行動の確認

### AI実行可能
- ログ・アクセス数の確認
- エラーログの確認
- 利用率・アクティブ率

### 人間アクション候補（出力のみ）
- ユーザーの声を3件以上収集
- サポート問い合わせの傾向確認
- 観察（ユーザーが実際にどう使っているか）

## Phase 3: 学習抽出

### 判定

| 判定 | 基準 |
|------|------|
| **HIT** | 仮説通り、またはそれ以上の効果 |
| **MISS** | 仮説と実測にギャップあり |
| **UNKNOWN** | データ不足で判定不能 |

### MISSの場合の深掘り

問題は3層のどこにあるか:

```
WHO層: ターゲットがそもそも違う？
WHAT層: 価値提案がズレてる？
HOW層: 実装（UI/UX、性能）の問題？
```

- WHO層の問題 → product-conviction で戦略を再検討
- WHAT層の問題 → spec-review で仕様を再検討
- HOW層の問題 → quality-scan で品質を改善

## Phase 4: LEARN_LOG 記録

`.claude/LEARN_LOG.md` に以下を記録:
- 日付、機能名、判定（HIT/MISS/UNKNOWN）
- 学習内容（1〜3行）
- 次アクション

## Phase 5: フィードバックをプロダクトに戻す

- P0優先順位の変更が必要か？
- 改善タスクの追加が必要か？
- 設計書の前提が変わっていないか？

## 完了条件

- [ ] spec.md の Business Impact Hypothesis と実測値を比較
- [ ] HIT / MISS / UNKNOWN 判定
- [ ] MISSの場合、問題の層（WHO/WHAT/HOW）を特定
- [ ] LEARN_LOG に記録済み
- [ ] 次アクションを決定
